## Description

**DisasterGuard** is a rule-based language interpreter designed for natural disaster detection and response. The system allows users to define declarative rules of the form `if <condition> then <action>`, which are evaluated over an environment of sensor variables and active facts. When conditions are met, new facts are activated, potentially triggering further rules in a chain reaction that models real-world emergency escalation protocols.

The project implements the full pipeline of a small compiler:

- **Lexical analysis** — tokenizes the input program
- **Syntactic analysis (LL(1))** — validates grammar and builds an AST using recursive descent
- **Semantic interpretation** — evaluates rules using a fixed-point execution model
- **Static analysis** — detects conflicts, redundancies, and potentially inactive rules

### Domain

The language is applied to a **natural disaster monitoring scenario**, where sensor readings (seismic magnitude, rainfall, river levels, coastal proximity, etc.) trigger escalating emergency protocols such as evacuation orders, shelter activation, national emergency broadcasts, and requests for military or air support.

### Language Specification

A program consists of two sections:

**Rules section:**
```
rule <id>:
if <condition> then <action>
```

**State section:**
```
State:
variable = integer
active_fact
```

Allowed conditions:
- Arithmetic comparison: `id > n`, `id < n`, `id = n`
- Active fact check: `id`
- Logical conjunction: `condition AND condition`

### Example Program

```
rule seismic_alert:
if magnitude > 6 then earthquake_detected

rule tsunami_risk:
if earthquake_detected AND coastal_distance < 50 then tsunami_warning

rule coastal_evacuation:
if tsunami_warning AND population > 500 then evacuate_coastal

State:
magnitude = 7
coastal_distance = 30
population = 1200
```

**Output:**
```
earthquake_detected
evacuate_coastal
tsunami_warning
```

## How to Run

### Requirements

| Tool | Version used |
|------|-------------|
| Operating System | Windows 11 |
| C++ Compiler | g++ 13.2.0 (MinGW-w64) |
| C++ Standard | C++17 |

To verify your compiler is available before compiling:
```powershell
g++ --version
```

If `g++` is not recognized, install it via [MSYS2](https://www.msys2.org/) or [MinGW-w64](https://www.mingw-w64.org/).

---

### Step 1 — Navigate to the project folder

Open a terminal (PowerShell or CMD) and navigate to the project directory:

```powershell
cd Formal-Languages-Final-Assignment
cd files
```

All source files are located here:
```
files/
├── main.cpp        # Complete source code — all 8 modules
├── README.md       # This file
└── disaster.txt    # Example input file
```

---

### Step 2 — Compile

Run the following command inside the `files/` folder:

```powershell
g++ -std=c++17 -Wall -O2 -o disasterguard main.cpp
```

This produces `disasterguard.exe` in the same folder. No additional tools or build systems are required.

---

### Step 3 — Run

**From a `.txt` file (recommended):**
```powershell
.\disasterguard disaster.txt
```

**From standard input (type directly in the terminal):**
```powershell
.\disasterguard
```
Type the rules and state, then press `Ctrl+Z` followed by `Enter` to signal end of input.

**Verbose mode** — shows a formal trace of each pipeline phase:
```powershell
.\disasterguard -v disaster.txt
```
The trace is printed to `stderr` and does not affect the canonical output on `stdout`.

---

### Input File Format

```
rule <rule_id>:
if <condition> then <action_id>

State:
sensor_variable = integer_value
initial_active_fact
```

| Rule | Detail |
|------|--------|
| Keywords `rule`, `if`, `then` | Lowercase, case-sensitive |
| Keyword `AND` | Uppercase, case-sensitive |
| Identifiers | Lowercase letters, digits, underscores — must start with a letter |
| Integer values | Non-negative base-10 numbers |
| `State:` section | Optional — separates rules from the initial environment |
| Whitespace and blank lines | Ignored |

---

### Output Format

The program prints, in order:

1. **Activated facts** — one per line, in alphabetical order. Only facts activated *during execution* are shown; initial state facts are excluded.
2. **`(no output)`** — if no new facts were activated.
3. **Static analysis messages** — printed after execution output:

| Type | Format |
|------|--------|
| Conflict | `Action <id> generated by r1, r2, ...` |
| Redundancy | `Redundant rules: r1, r2` |
| Potentially inactive | `Potentially inactive rule: <id>` |

## Algorithm Explanations

### Module 1 — Tokens

Defines the `TokenType` enum and `Token` struct. Every meaningful unit of the source text is represented as a token with a type, its original text (`lexeme`), and the source line number (used for error reporting).

Recognized token types: `RULE`, `IF`, `THEN`, `AND`, `COLON`, `GT`, `LT`, `EQ`, `ID`, `VALUE`, `END_OF_FILE`.

---

### Module 2 — Lexical Analyzer (Lexer)

The `Lexer` class reads the source character by character and produces a flat list of tokens. It skips whitespace and blank lines, distinguishes keywords from identifiers, and discards unrecognized words.

Key rules:
- Identifiers match `[a-z_][a-z0-9_]*`
- Keywords (`rule`, `if`, `then`) are recognized after reading an identifier
- `AND` is recognized separately as it starts with an uppercase letter
- Integer literals match `[0-9]+`

---

### Module 3 — Abstract Syntax Tree (AST)

Defines the node hierarchy used to represent programs in memory:

```
ASTNode (base)
  |- FactNode    — condition: fact is active (id)
  |- CmpNode     — condition: comparison (id op value)
  |- AndNode     — condition: conjunction (left AND right)
  +- RuleNode    — rule: name + condition + action
ProgramNode      — list of RuleNode
```

Smart pointers (`shared_ptr`) are used throughout to manage node lifetimes safely.

---

### Module 4 — LL(1) Syntactic Analyzer (Parser)

The `Parser` class implements a **recursive descent parser**, where each grammar production is directly encoded as a method. Production selection is made by inspecting the current lookahead token (1-symbol lookahead).

The original grammar from the project specification:
```
Cond -> Cond AND Cond | Atom
```
is **left-recursive** and therefore not LL(1). As proven in exercise 4.16 of Aho et al., no left-recursive grammar can be LL(1). It was transformed by eliminating left recursion:

```
Program  -> RuleList
RuleList -> Rule RuleList | e
Rule     -> rule id : if Cond then Action
Cond     -> Atom Cond'
Cond'    -> AND Atom Cond' | e
Atom     -> id Atom'
Atom'    -> RelOp value | e
RelOp    -> > | < | =
Action   -> id
```

**FIRST and FOLLOW sets:**

| Non-terminal | FIRST          | FOLLOW        |
|---|---|---|
| `RuleList`   | `{rule, e}`    | `{$}`         |
| `Cond`       | `{id}`         | `{then}`      |
| `Cond'`      | `{AND, e}`     | `{then}`      |
| `Atom'`      | `{>, <, =, e}` | `{AND, then}` |

The parser constructs the AST directly during parsing (single pass).

---

### Module 5 — Interpreter (Fixed-Point Semantics)

The `Interpreter` class evaluates the AST over a `State` consisting of:
- `vars`: a mapping from identifiers to integer values
- `facts`: a set of currently active facts

**Condition evaluation:**
- `CmpNode(x, op, v)` — true if `vars[x] op v`
- `FactNode(x)` — true if `x` is in `facts`
- `AndNode(l, r)` — true if `eval(l)` and `eval(r)` (short-circuit)

**Fixed-point execution model:**
```
S0 = initial state
Si+1 = Si U {a | exists rule r in P : cond(r) is true in Si}
S* = Sn when Sn = Sn-1   (convergence — point fix reached)
```

The algorithm is guaranteed to terminate because the fact set is finite and facts are only ever added, never removed.

---

### Module 6 — Static Analyzer

Performs three independent analyses over the AST **without executing the program**:

**1. Conflict detection**
Groups rules by their action identifier. If two or more rules produce the same action, a conflict is reported.

**2. Redundancy detection**
Two rules are redundant if their conditions and actions are structurally identical. AND operands are sorted alphabetically before comparison so that `a AND b` and `b AND a` are treated as equal.

**3. Potentially inactive rule detection**
Computes the set of reachable identifiers via forward propagation from the initial state. Starting from all variables and initial facts, the algorithm propagates: if all identifiers in a rule's condition are reachable, its action becomes reachable too. A rule is potentially inactive if any identifier in its condition never becomes reachable.

---

### Module 7 — Initial State Parser

Parses the `State:` section as plain text (independent of the main lexer). Each non-empty line is classified as:
- `id = integer` — variable assignment added to `vars`
- `id` — active fact added to `facts`

---

### Module 8 — Main

Orchestrates the full pipeline:
1. Read input from file argument or standard input
2. Split content on `State:` to separate rules from state
3. Run the lexer on the rules section
4. Run the parser to build the AST
5. Parse the initial state
6. Run the interpreter and collect activated facts
7. Print execution output in alphabetical order
8. Run the static analyzer and print messages
