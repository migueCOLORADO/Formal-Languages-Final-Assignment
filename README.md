## Description

**DisasterGuard** is a rule-based language interpreter designed for natural disaster detection and response. The system allows users to define declarative rules of the form `if <condition> then <action>`, which are evaluated over an environment of sensor variables and active facts. When conditions are met, new facts are activated, potentially triggering further rules in a chain reaction that models real-world emergency escalation protocols.

The project implements the full pipeline of a small compiler:

- **Lexical analysis** — tokenizes the input program
- **Syntactic analysis (LL(1))** — validates grammar and builds an AST using recursive descent
- **Semantic interpretation** — evaluates rules using a fixed-point execution model
- **Static analysis** — detects conflicts, redundancies, and potentially inactive rules

### Domain

The language is applied to a **natural disaster monitoring scenario**, where sensor readings (seismic magnitude, rainfall, river levels, coastal proximity, etc.) trigger escalating emergency protocols such as evacuation orders, shelter activation, national emergency broadcasts, and requests for military or air support.

### Language Specification

A program consists of two sections:

**Rules section:**
```
rule <id>:
if <condition> then <action>
```

**State section:**
```
State:
variable = integer
active_fact
```

Allowed conditions:
- Arithmetic comparison: `id > n`, `id < n`, `id = n`
- Active fact check: `id`
- Logical conjunction: `condition AND condition`

### Example Program

```
rule seismic_alert:
if magnitude > 6 then earthquake_detected

rule tsunami_risk:
if earthquake_detected AND coastal_distance < 50 then tsunami_warning

rule coastal_evacuation:
if tsunami_warning AND population > 500 then evacuate_coastal

State:
magnitude = 7
coastal_distance = 30
population = 1200
```

**Output:**
```
earthquake_detected
evacuate_coastal
tsunami_warning
```

## How to Run

### Requirements

| Tool | Minimum Version |
|------|----------------|
| Operating System | Linux, macOS, or Windows (WSL) |
| C++ Compiler | g++ >= 7.0 with C++17 support |
| Build Tool | GNU Make |

Verify your compiler version:
```bash
g++ --version
```

### Compilation

Using Make (recommended):
```bash
make
```

Or manually:
```bash
g++ -std=c++17 -Wall -Wextra -O2 -o disasterguard main.cpp
```

### Execution

**From a file:**
```bash
./disasterguard tests/disaster.txt
```

**From standard input:**
```bash
./disasterguard < tests/disaster.txt
```

**Run all test cases:**
```bash
make test
```

### Input File Format

```
rule <rule_id>:
if <condition> then <action_id>

State:
sensor_variable = integer_value
initial_active_fact
```

- Rules and the state section are separated by the `State:` keyword.
- Rules are separated from each other by blank lines.
- Keywords `rule`, `if`, `then` are lowercase; `AND` is uppercase — both are case-sensitive.
- Identifiers consist of lowercase letters, digits, and underscores, and must start with a letter.
- Integer values are non-negative base-10 numbers.
- Extra whitespace and blank lines are ignored.

### Output Format

The program prints, in order:

1. **Activated facts** — one per line, in alphabetical order. Only facts activated *during execution* are printed; initial state facts are excluded.
2. **`(no output)`** — printed if no new facts were activated.
3. **Static analysis messages** — printed after execution output:
   - `Action <id> generated by r1, r2, ...` — conflict
   - `Redundant rules: r1, r2` — redundancy
   - `Potentially inactive rule: <id>` — inactive rule

### Test Cases

| File | Description | Expected output |
|------|-------------|-----------------|
| `tests/case1.txt` | Basic seismic alert | `earthquake_detected` |
| `tests/case2.txt` | Three-rule chain reaction | `earthquake_detected`, `evacuate_now`, `tsunami_risk` |
| `tests/case3.txt` | Threshold not reached | `(no output)` |
| `tests/case4.txt` | Two rules, same action | `emergency_alert` + conflict message |
| `tests/case5.txt` | Identical rules | `flood_alert` + redundancy message |
| `tests/disaster.txt` | Full scenario (10 rules) | 9 activated facts + 1 inactive rule message |

## Algorithm Explanations

### Project Structure

```
DisasterGuard/
├── main.cpp          # Complete implementation — 8 internal modules
├── Makefile          # Build configuration
├── README.md         # This file
└── tests/
    ├── case1.txt
    ├── case2.txt
    ├── case3.txt
    ├── case4.txt
    ├── case5.txt
    └── disaster.txt
```

All logic is contained in `main.cpp`, organized into 8 clearly delimited modules following a modular design principle.

---

### Module 1 — Tokens

Defines the `TokenType` enum and `Token` struct. Every meaningful unit of the source text is represented as a token with a type, its original text (`lexeme`), and the source line number (used for error reporting).

Recognized token types: `RULE`, `IF`, `THEN`, `AND`, `COLON`, `GT`, `LT`, `EQ`, `ID`, `VALUE`, `END_OF_FILE`.

---

### Module 2 — Lexical Analyzer (Lexer)

The `Lexer` class reads the source character by character and produces a flat list of tokens. It skips whitespace and blank lines, distinguishes keywords from identifiers, and discards unrecognized words (such as `State`, which is handled before the lexer is invoked).

Key rules:
- Identifiers match `[a-z_][a-z0-9_]*`
- Keywords (`rule`, `if`, `then`) are recognized after reading an identifier
- `AND` is recognized separately as it starts with an uppercase letter
- Integer literals match `[0-9]+`

---

### Module 3 — Abstract Syntax Tree (AST)

Defines the node hierarchy used to represent programs in memory:

```
ASTNode (base)
  ├── FactNode    — condition: fact is active (id)
  ├── CmpNode     — condition: comparison (id op value)
  ├── AndNode     — condition: conjunction (left AND right)
  └── RuleNode    — rule: name + condition + action
ProgramNode       — list of RuleNode
```

Smart pointers (`shared_ptr`) are used throughout to manage node lifetimes safely.

---

### Module 4 — LL(1) Syntactic Analyzer (Parser)

The `Parser` class implements a **recursive descent parser**, where each grammar production is directly encoded as a method. No parsing table is constructed explicitly; the production choice is made by inspecting the current lookahead token (1-symbol lookahead).

The original grammar from the project specification:
```
Cond -> Cond AND Cond | Atom
```
is **left-recursive** and therefore not LL(1). It was transformed by eliminating left recursion and applying left factoring:

```
Program  -> RuleList
RuleList -> Rule RuleList | e
Rule     -> rule id : if Cond then Action
Cond     -> Atom Cond'
Cond'    -> AND Atom Cond' | e
Atom     -> id Atom'
Atom'    -> RelOp value | e
RelOp    -> > | < | =
Action   -> id
```

**FIRST and FOLLOW sets (relevant entries):**

| Non-terminal | FIRST        | FOLLOW       |
|---|---|---|
| `RuleList`   | `{rule, e}`  | `{$}`        |
| `Cond`       | `{id}`       | `{then}`     |
| `Cond'`      | `{AND, e}`   | `{then}`     |
| `Atom'`      | `{>, <, =, e}` | `{AND, then}` |

The parser constructs the AST directly during parsing (single pass).

---

### Module 5 — Interpreter (Fixed-Point Semantics)

The `Interpreter` class evaluates the AST over a `State` consisting of:
- `vars`: a mapping from identifiers to integer values
- `facts`: a set of currently active facts

**Condition evaluation** is recursive over the AST:
- `CmpNode(x, op, v)` — true if `vars[x] op v`
- `FactNode(x)` — true if `x` is in `facts`
- `AndNode(l, r)` — true if `eval(l)` and `eval(r)` (short-circuit)

**Execution model (fixed-point):**
```
1. Initialize facts with the initial state
2. Repeat:
   a. Evaluate all rules
   b. For each rule whose condition is true and action not yet active:
      add action to facts, mark as changed
3. Until no new facts are added (convergence)
```

The algorithm is guaranteed to terminate because the set of possible facts is finite and facts are only ever added, never removed. Only facts activated *during execution* are included in the output; facts present in the initial state are excluded.

---

### Module 6 — Static Analyzer

Performs three independent analyses over the AST without executing the program:

**1. Conflict detection**
Groups rules by their action identifier. If two or more rules share the same action, a conflict is reported.

**2. Redundancy detection**
Two rules are redundant if their conditions and actions are structurally identical. Conditions are normalized to a canonical string (AND operands are sorted alphabetically) to detect semantic equality regardless of operand order.

**3. Potentially inactive rule detection**
Computes the set of *reachable identifiers* starting from the initial state (variables and initial facts) and propagating forward through rules whose conditions are satisfiable. A rule is considered potentially inactive if any identifier in its condition is never reachable under this forward analysis.

---

### Module 7 — Initial State Parser

Parses the `State:` section as plain text. Each non-empty line is classified as:
- `id = integer` — variable assignment
- `id` — active fact

---

### Module 8 — Main

Orchestrates the full pipeline:
1. Read input from file or stdin
2. Split on `State:` to separate rules from state section
3. Lex the rules section
4. Parse tokens into AST
5. Parse initial state
6. Run interpreter and collect activated facts
7. Print execution output (alphabetically sorted)
8. Run static analyzer and print messages
